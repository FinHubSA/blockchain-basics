Pathfinding on the XRPL DEX

What Is Pathfinding?

On the XRP Ledger, paths describe how a payment can flow from a sender to a receiver, potentially across multiple currencies, issuers, and liquidity sources. Pathfinding is the process of discovering those paths so a single Payment transaction can deliver the desired amount, even when the sender and receiver do not hold the same currency directly (paths concept).

Key idea: Paths connect trust lines, order books, and AMM pools so a cross‑currency payment can be executed atomically in one transaction.

How Paths Use the DEX

A path is made up of steps. Each step can either:

  • Ripple through an account on a trust line (moving balances of the same token between two holders), or
  • Trade through the DEX, using an order book or AMM to change from one currency to another.

When a Payment transaction executes, XRPL can combine liquidity from multiple DEX sources (order books and AMMs) to deliver the destination amount, choosing the best available prices at the time the ledger is validated.

Path Sets and Default Paths

A transaction can specify a path set: a collection of one or more candidate paths the payment is allowed to use. In addition, XRPL can consider a default path implied by the transaction’s fields, such as Account, Destination, Amount, and SendMax.

  • If the source and destination currencies are the same, the default path may simply ripple balances across trust lines.
  • For cross‑currency payments, the default path typically uses the best available order book or AMM between the source and destination currencies.
  • The tfNoDirectRipple flag can disable the default path so only explicitly provided paths are allowed.

Pathfinding APIs

The core XRPL server (rippled) exposes dedicated API methods for pathfinding:

  • ripple_path_find: one‑shot lookup of possible path sets between a source and destination.
  • path_find (WebSocket only): a streaming version that keeps updating as new ledgers close and better paths appear.

You can also have rippled autofill paths when signing a Payment by setting build_path, but best practice is to run pathfinding explicitly, review the results, and then sign and submit the transaction.

Important: Pathfinding aims to find good paths, not mathematically perfect ones. The available liquidity and prices can change slightly between path discovery and final ledger validation, and untrusted servers could theoretically bias results. Always validate assumptions and use reputable XRPL infrastructure.

When to Use Paths

  • Cross‑currency payments: Pay a destination in their preferred stablecoin or token while you send a different asset.
  • Tokenized FX and routing: Move value through multiple order books and AMMs in one hop.
  • Community credit and complex IOU networks: Settle multi‑party debts by rippling balances through shared issuers.

For a deeper dive into how paths are modeled and serialized, including path step fields and default path rules, refer to the official XRPL Paths documentation.
