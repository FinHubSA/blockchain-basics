XRPL Consensus & Nodes

How the XRP Ledger Reaches Consensus

The XRP Ledger does not use Proof of Work or Proof of Stake mining. Instead it uses the XRP Ledger Consensus Protocol: a group of validators repeatedly run a consensus round to agree on the set and order of transactions, then close the ledger. A new ledger typically closes every 3–5 seconds.

Types of Nodes in the XRPL Ecosystem

Different nodes play different roles in the network.

Validators

  • Validator node: Runs the consensus protocol and votes on which transactions to include and the order of ledger closes. Validators are chosen by others (e.g. through Unique Node Lists) and need to be trusted for the health of the network.
  • Validator list (UNL): Each node configures a Unique Node List — a set of validators it trusts. Consensus is reached among the validators on your UNL; the network works as long as enough participants overlap in trust.

Non-validating nodes

  • Full node (rippled): Runs the same software as validators and stores a full copy of the ledger but does not participate in consensus. Used for RPC, history, and reliability.
  • Reporting mode: A lighter mode that follows the network and serves API requests without storing full history. Good for applications that need current state and recent data.

Other roles

  • P2P / cluster: Nodes communicate over peer-to-peer links to propagate transactions and proposed ledgers. Clustering helps with redundancy and performance.
  • API / RPC: Many users and applications connect to public or private API nodes (e.g. xrpl.org, Ripple, or self-hosted) instead of running their own node.
In practice: To send transactions you typically use a client library (e.g. xrpl.js) and connect to a public or private node; you don’t need to run a validator or full node yourself.
