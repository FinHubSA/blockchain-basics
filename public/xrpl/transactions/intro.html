XRPL Transactions

What is an XRPL Transaction?

A transaction on the XRP Ledger is a signed instruction that changes ledger state: sending XRP, creating offers, setting trust lines, escrows, and more. Each transaction type has a defined format and is signed by the account’s secret key.

Signing: As in the Transaction Signing section of the Ethereum tutorial, only the holder of the private key can produce a valid signature. XRPL uses the same elliptic curve (secp256k1) with a different serialization and encoding.

Common Transaction Fields

Every XRPL transaction includes a set of common fields that identify the sender, prevent replay, and set cost and expiry:

  • Account: The classic address (e.g. r...) of the sender. This account’s key pair must sign the transaction.
  • Fee: The amount of XRP (in drops; 1 XRP = 1,000,000 drops) paid for the transaction. This XRP is destroyed (see Fees and burning below).
  • Sequence: A per-account counter that increments with each transaction. It enforces ordering and prevents replay: the same transaction cannot be applied twice.
  • LastLedgerSequence: (Optional.) The last ledger index in which this transaction can be included. If the ledger has advanced past this index, the transaction is invalid. Use it to avoid leaving transactions “in flight” indefinitely.
  • SigningPubKey: The public key used to verify the signature. Empty for multi-signed transactions, where Signers is used instead.
  • TxnSignature: The signature over the canonical transaction bytes, proving the sender authorizes the transaction.
  • Memos: (Optional.) An array of memo objects that attach arbitrary data to the transaction. Each memo can include MemoData (hex-encoded payload), and optionally MemoType and MemoFormat (also hex). Memos are not interpreted by the ledger—they are stored on-chain for reference. Common uses include payment references, invoice IDs, notes to the recipient, or compliance tags. Because memo data is visible on the ledger, avoid putting sensitive or personally identifiable information in memos.

Transaction-type-specific fields (e.g. Destination, Amount for Payment) are added on top of these.

Common Transaction Types

  • Payment: Send XRP or issued currencies to another account.
  • OfferCreate / OfferCancel: Create or cancel orders on the decentralized exchange.
  • TrustSet: Create or modify a trust line to hold issued currencies.
  • EscrowCreate / EscrowFinish / EscrowCancel: Time-locked or condition-based escrow.
  • AccountSet: Update account options (e.g. flags, domain).

Fees and the deflationary nature of XRP

On the XRP Ledger, the fee you pay (in XRP) is not paid to any validator or miner. Instead, that XRP is destroyed (burned): it is removed from the total supply and can never be used again. The minimum fee is a small amount of drops (currently 12 drops for a standard transaction, though it can rise with load). You can pay more than the minimum to prioritize your transaction when the network is busy.

Because fees are burned and no new XRP is minted for block production, the total supply of XRP is deflationary over time: as more transactions are processed, more XRP is destroyed. The initial supply was created at ledger inception; since then, the only change to supply is the decrease from burned fees. This contrasts with chains that issue new coins as block rewards, which can be inflationary unless offset by burning or other mechanisms.

Validator rewards: XRPL vs Ethereum

Ethereum: Validators (stakers) are rewarded for producing and attesting to blocks. They earn block rewards (newly issued ETH for each block) plus priority fees (tips) and, historically, MEV-related value. Validators are thus directly incentivized in ETH to run nodes and secure the network.

XRP Ledger: There are no block rewards and no XRP paid to validators for validating. Transaction fees are burned, not given to validators. Validators (university operators, exchanges, businesses, etc.) run nodes to participate in consensus and to have a trusted view of the ledger for their own use—not to earn XRP from the protocol. Incentives are indirect: a reliable, secure network benefits their services and users.

Summary: Ethereum rewards validators with newly issued ETH and fees; XRPL does not pay validators in XRP—fees are burned, making XRP deflationary and aligning validator motivation with operational need rather than block rewards.

Transactions are broadcast to the network and included in ledger closes by consensus (see Consensus Mechanisms).
