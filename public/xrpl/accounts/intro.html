XRPL Accounts

What is an XRPL Account?

An account on the XRP Ledger is an identity that can hold XRP, hold or issue other currencies, and submit transactions. Each account has a unique address (typically starting with r in X-address format or classic form) derived from the account’s public key.

Key derivation: XRPL uses the same elliptic curve (secp256k1) as Ethereum. The address is derived from the public key and encoded in a ledger-specific format (e.g. base58, X-address). You can generate keys in the Blockchain Concepts section and use them for XRPL with the appropriate encoding.

Who Controls an Account: Master, Regular, and Multi-Signature Keys

An account is controlled by the keys that can authorize transactions for it. Only signed transactions can be submitted and included in the ledger. The XRP Ledger supports three ways to authorize transactions for an account: the master key pair, a regular key pair, and multi-signing (a signer list). For more detail, see the official Cryptographic Keys documentation.

Master Key Pair

The master key pair is the key pair from which the account ID (address) is derived. The account’s classic address and X-address are tied to this public key. You cannot change or remove the master key pair; you can only disable it (and later re-enable it using a regular key or multi-signature).

Only the master key pair can authorize certain sensitive actions: sending the account’s first transaction, disabling the master key, permanently giving up the ability to freeze, and sending a special key reset transaction at 0 cost. For everything else, a regular key or multi-signature can act in place of the master key.

A common practice is to keep the master key pair offline (e.g. on an air-gapped device or in cold storage) and use a regular key pair for day-to-day signing. If the regular key is compromised, you can use the master key to set a new regular key and regain control without moving to a new address.

Regular Key Pair

An account can authorize a regular key pair—a secondary key pair that can sign most transactions the same as the master key. The regular key is stored in the AccountRoot’s RegularKey field and is set or changed with a SetRegularKey transaction. You can remove or replace the regular key at any time without changing the account’s address.

Regular keys are necessary when you want to sign from a hot wallet or an application while keeping the master key cold. They are used by specifying the regular key’s credentials when building and signing transactions; the ledger accepts the signature if it matches either the master or the current regular key.

Multi-Signature (Signer List)

An account can also use multi-signing: a list of signers (key pairs or other accounts) with weights and a threshold. A transaction is valid if the sum of the weights of the signers that signed it meets the threshold. This allows shared control (e.g. a treasury that requires 2-of-3 signers) or backup signers.

Multi-signature is necessary when multiple parties must agree before an action is taken, or when you want redundancy (e.g. several keys, any k of which can sign). It is used by submitting a transaction that includes signatures from enough signers to meet the threshold; the ledger verifies the combined authorization against the account’s signer list.

Summary: The master key pair defines the account’s address and can do a few things only it can do; it can be disabled but not changed. A regular key pair is an optional second key you set for daily use. Multi-signature lets multiple keys or accounts share control according to weights and a threshold. All three are ways to authorize transactions for the same account.

How Accounts Are Created

An account does not exist on the ledger until it has a ledger entry. On the XRP Ledger, account state is created by a Payment transaction: when someone sends enough XRP to a mathematically-valid address that does not yet have an account, the ledger creates a new account (an AccountRoot entry) for that address. The payment must meet the minimum reserve (and typically a bit more to leave the account with a usable balance). Until then, the address is just a possible destination; there is no AccountRoot and no balance.

AccountRoot Ledger Entry

Each account’s data is stored in an AccountRoot ledger entry. This is the ledger object that describes a single account, its settings, and its XRP balance. The main fields are:

  • Account – The classic address of the account (e.g. r...). This identifies the entry.
  • Balance – The account’s XRP balance in drops (1 XRP = 1,000,000 drops), stored as a string. Part of this is locked as the account reserve.
  • Sequence – The sequence number of the next valid transaction from this account. Each transaction must use the current sequence and then the account’s sequence increments by 1. This prevents replay and orders transactions.
  • OwnerCount – The number of ledger objects (trust lines, offers, escrows, etc.) owned by this account. Used to compute the owner reserve (additional XRP that must be held).
  • Flags – A bitmask of account options (e.g. default ripple, deposit auth, require destination tag, disable master key). Many can be changed with an AccountSet transaction.
  • PreviousTxnID – The hash of the transaction that last modified this AccountRoot.
  • PreviousTxnLgrSeq – The ledger index that contains that transaction.
  • RegularKey – (Optional) Another address that can sign for this account instead of the master key.
  • Domain, EmailHash, MessageKey – (Optional) Metadata and messaging hints.

The AccountRoot entry type is fixed when the account is created; only its fields (e.g. Balance, Sequence, Flags) change as transactions are applied. You can look up an account’s state via APIs such as account_info.

Account Properties

  • Reserve: A small XRP reserve is required to exist on the ledger (to prevent spam).
  • Sequence: Each transaction consumes a sequence number to prevent replay.
  • Signers: Accounts can use multi-signing for shared control.

Create or Import an Account

You can create or import an XRPL account using a family seed (secret) or a hex private key. The tool uses the xrpl library to derive your classic address and public key.
